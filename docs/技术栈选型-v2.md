# 鼎伟 CRM 业务人员端 — 技术栈选型决策书 v2

**版本**:v2.0
**日期**:2026-05-20
**状态**:Phase 0 产出,推翻 v1
**前版**:`docs/技术栈选型.md` v1 (归档,仅作历史参考)

---

## 1. 为什么 v1 被推翻

v1 选 NestJS + PostgreSQL + Redis 的栈,看似主流,但**没有附决策依据**,只是堆砌词汇。访谈中暴露三个 v1 没考虑的硬约束:

1. **现有生产系统数据库是 SQL Server**——选 Postgres 起步会被后期方言迁移卡住。
2. **团队同事主用 Java / Python**——NestJS/TypeScript 会让团队接手维护别扭,长期承担成本。
3. **AI 生态在 Python 这边**——DeepSeek SDK、langchain、视觉模型工具链(CLIP/SigLIP2/DINOv2)、Milvus 等几乎全是 Python first,选 Node 会让 AI 路径承担额外的语言桥接成本。

v2 选型从这三个硬约束反推。

## 2. 选型一览(每条附决策依据)

| 层 | 选型 | 决策依据 |

## 3. 被否决的方案及原因

### 3.1 Node.js / NestJS (v1 选择)

**否决理由**:团队主语言是 Java / Python,Node 团队接手成本高。AI 生态(DeepSeek SDK、视觉模型、Milvus client)在 Python 更完善,Node 路径需要语言桥接。

### 3.2 Django

**否决理由**:Django 的最大价值是 django-admin + Django ORM + 表单/auth 一体化。但本项目:
- 老板端是独立系统,不复用 django-admin。
- Pydantic + SQLAlchemy 比 Django ORM 在 SQL Server 上更灵活(尤其 native_enum=False 的 CHECK 约束写法)。
- Django 同步路由对 LLM 长响应不友好(WSGI vs ASGI),要走 Django Channels 又是一套学习成本。

如果团队明确希望 Django 落地(因为 Django 老师傅多),v3 可重新评估。

### 3.3 Java / Spring Boot

**否决理由**:不是反对 Java——而是这个项目 AI 占核心比重,Spring AI 生态相比 Python 还在追赶,DeepSeek Java SDK 也不如 Python 完善。同时 Spring Boot 启动慢 / 调试慢的特性对 1-2 月试点节奏不利。**未来如果做老板端管理后台,Spring Boot 是合理选择**——但业务人员端这种 AI 重的应用 Python 更顺。

### 3.4 PostgreSQL (v1 选择)

**否决理由**:见 §1。要避免与现有 SQL Server 系统的方言不一致带来后期迁移成本。Postgres 强在 JSONB / 类型扩展 / 数组等,但本项目数据模型纯关系型,SQL Server 完全够用。

### 3.5 Prisma (v1 隐含)

**否决理由**:Prisma 是 Node 生态的 ORM,不在 Python 选型范围内。

### 3.6 MongoDB / Elasticsearch / 其他 NoSQL

**否决理由**:本项目数据模型纯关系型,有外键约束、有事务需求(拜访 + 客户最后拜访时间在同事务),不适合 NoSQL。AI 找板的向量数据走 Milvus(专用向量库),不是通用 NoSQL。

## 4. SQL Server 在本项目里的工程化坑(给后续开发者的提醒)

1. **Mac M 系列(arm64)**:`mcr.microsoft.com/mssql/server:2022-latest` 官方只支持 amd64。docker-compose 需要 `platform: linux/amd64`,本机先 `softwareupdate --install-rosetta`。或换 `azure-sql-edge`(ARM 友好但部分新特性缺失,语法上 99% 兼容)。
2. **ODBC 驱动安装链**:Mac 本机非 docker 跑后端需要 `brew tap microsoft/mssql-release` 然后 `brew install msodbcsql18 mssql-tools18`,首次坑较多。**首选在 docker 内运行 backend**,Mac 本机只在 IDE 调试时直连。
3. **没有原生 ENUM**:用 `VARCHAR(32)` + CHECK 约束(SQLAlchemy `Enum(..., native_enum=False)`)。Alembic 迁移修改 enum 值时要手动写 raw SQL 改 CHECK,不像 Postgres 一条 `ALTER TYPE`。
4. **多路 CASCADE 限制**:`ON DELETE CASCADE / SET NULL` 在 SQL Server 不允许形成多路径,会报 "may cause cycles or multiple cascade paths"。本方案只在 `Plan.customer_id` 用 SetNull,其余 `RESTRICT`/`NO ACTION`。
5. **DATETIME2 vs DATETIME**:用 `DATETIME2(7)`,精度高于老 `DATETIME`,与 Python `datetime` 微秒级映射兼容。`server_default=func.sysutcdatetime()`(SQL Server 的 UTC 时间函数)。
6. **UNIQUEIDENTIFIER 主键**:`uuid_generate_v4()` 在 SQL Server 是 `NEWID()`,在 SQLAlchemy 用 `Uuid(as_uuid=True)` + Python 侧 `default=uuid.uuid4` 而非依赖 server-side default(避免与 returning 子句的 SQL Server 实现差异)。
7. **Identity column 与 GUID 并存**:本项目所有主键用 UNIQUEIDENTIFIER,不混用 INT IDENTITY,避免某些表的 ID 是数字、某些是 GUID 的混乱。
8. **Alembic 自动生成迁移的 SQL Server 兼容性**:`alembic revision --autogenerate` 在 SQL Server 上对 ENUM 类型变更生成的 DDL 不准,**重要变更必须手工检查迁移文件**。

## 6. 未来重评估的触发条件

如下任一条件出现时,v2 选型应重新评估:
1. 团队语言重心从 Python/Java 漂移到其他栈。
2. 老板端系统的 DB 不再是 SQL Server。
3. AI 找板需求被取消(Python 生态优势减弱)。
4. 用户量从中型(50 同时在线)突破到大型(几百同时在线),需要更激进的扩展架构。
5. 公司增加 Java 占比超过 70% 且 AI 工作量被独立微服务化(可考虑业务主线 Spring Boot + AI 微服务 FastAPI)。
