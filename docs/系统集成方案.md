# 系统集成方案 — 业务人员端 ↔ 老板端 ↔ AI 服务

**版本**:v1.0(本文档为新增,无前版)
**日期**:2026-05-20
**状态**:Phase 0 产出,描述独立库 + 后续同步形态的具体落地方式

---

## 1. 集成边界总览

业务人员端不直接读写老板端的 SQL Server 实例。所有跨系统交互通过 **REST API + 字段级映射**完成。

```
┌─────────────────────┐         ┌──────────────────────┐
│  业务人员端         │         │  老板端(重构中)    │
│  (Python/FastAPI)   │◀───────│  (栈待定)            │
│  独立库             │  REST   │  SQL Server 主库     │
│  dingwei_crm_sales  │───────▶│  (客户/订单/样板)    │
└──────┬──────────────┘         └──────────────────────┘
       │
       │ Function Calling
       ▼
┌─────────────────────┐         ┌──────────────────────┐
│  DeepSeek API       │         │  公司视觉服务         │
│  (文本 LLM)         │         │  (SigLIP2+DINOv2)    │
│                     │         │  Milvus 向量库       │
└─────────────────────┘         └──────────────────────┘
```

**核心原则**:

1. **业务人员端有自己的 schema 和库**——不直接 SELECT 老板端的表。这避免被老板端历史结构绑死,也避免重构期双方耦合卡进度。
2. **预留 external_id**——所有跨系统资源(user/customer/sample/order)都有 `external_id` + `source_system` + `synced_at` 三个字段,后期同步无需 schema 迁移。
3. **数据流向单一**——客户主数据从老板端单向流入;业务人员端的拜访/计划/周报数据单向回流给老板端。不做双向同步,避免冲突解决复杂度。

---

## 2. 数据所有权矩阵

| 数据 | Source of Truth | 流向 | 备注 |
|---|---|---|---|
| 客户主数据(`customer`) | **老板端** | 老板端 → 业务人员端 | 客户名称、联系人、地址等基本信息由老板端管理。业务人员端只读副本。 |
| 客户分级(A/B/C) | **业务人员端**(本端 AI 估评 + 销售 override) | 可回传老板端 | 老板端可读不可改。 |
| 销售人员账号 | **悬挂**(待与老板端对齐) | 双方共享 | 见 §6 账号体系。 |
| 拜访计划(`plan`,业务计划) | **业务人员端** | 业务人员端 → 老板端 | `is_personal=False` 的回传;`is_personal=True` 个人计划不回传。 |
| 拜访记录(`visit_record`) | **业务人员端** | 业务人员端 → 老板端 | 全量回传。 |
| 周报(`weekly_report`) | **业务人员端** | 业务人员端 → 老板端 | 仅 `status='submitted'` 的回传。 |
| 样板(`sample`) | **老板端** | 老板端 → 业务人员端 | 业务人员端只读。后续 AI 找板用本地副本 + Milvus 向量。 |
| 订单(`order`) | **老板端** | 老板端 → 业务人员端 | 1A 不接入,Phase 2 通过 API 实时查询(不落本地副本)。 |
| 样板图片 | **公司视觉服务** | 不进业务人员端 DB | 几万张样板图在公司视觉服务的对象存储;业务人员端通过 sample.image_url + Milvus query 引用,不下载到本地。 |

## 3. 同步策略(Phase 2 实施,1A 仅留 schema 字段)

### 3.1 客户主数据(老板端 → 业务人员端)

**方案**:**老板端提供 REST API,业务人员端定时拉取**(由访谈确认)。

**API 契约示例**(待老板端团队对齐):

```http
GET https://laoban-api.dingwei.internal/api/v1/customers?updated_since={iso8601}
Authorization: Bearer {service-account-token}

Response 200:
{
  "items": [
    {
      "external_id": "CUST-20231024-0001",
      "name": "宏远服饰",
      "short_name": "宏远",
      "contact_name": "李总",
      "contact_title": "总经理",
      "phone": "13800006271",
      "address": "佛山南海区里水镇宏远工业园",
      "updated_at": "2026-05-20T08:30:00+08:00"
    }
  ],
  "next_cursor": "..."
}
```

**业务人员端拉取逻辑**(Phase 2 实施):

- 后台任务(Celery + Redis):每 30 分钟拉一次,按 `synced_at` 增量。
- Upsert 按 `external_id` 匹配。已存在则更新基本信息字段,**不覆盖** `owner_id`、`level`、`ai_score`、`tags`(这些是业务人员端管理的);新客户写入时按公司分配规则选 owner_id(可由老板端在拉取响应里指定 `recommended_owner_external_id`,业务人员端按 `user.external_id` 匹配)。
- 删除策略:老板端删除客户时本端 soft delete(`customer.status='inactive'`),不物理删,避免拜访历史断链。

### 3.2 业务人员端产生数据 → 老板端

**方案**:**业务人员端提供 REST API,老板端定时拉取 OR 业务人员端定时推送**(2 选 1,Phase 2 与老板端团队定)。

如果老板端拉(推荐——主动权在数据消费方):

```http
GET https://yewuyuan-api.dingwei.internal/api/v1/integration/visit-records?updated_since={iso8601}
Authorization: Bearer {service-account-token}

Response: 标准拜访记录列表 + cursor
```

如果业务人员端推:

- Celery 任务每小时一次扫 `visit_record.synced_at IS NULL` 的记录,POST 到老板端,成功后更新 `synced_at`。

**回传过滤**:

- `plan.is_personal=True` 的项目 **不回传**。
- `weekly_report.status='draft'/'reopened'` 不回传,仅 `submitted` 回传。
- 业务人员端的客户分级(A/B/C)、AI 估评分数、自定义 tags 可回传,但老板端不能反向覆盖。

### 3.3 样板数据(老板端 → 业务人员端)

**1A**:不接入,仅 schema 字段。

**Phase 2**:与样板视觉服务同期上线,两路并行:
- 样板元数据(sample_no、宽度、开力度、状态、image_url)从老板端拉,落 `sample` 表。
- 样板的向量 embedding 由公司视觉服务计算,落 Milvus collection,`sample.embedding_ref` 字段存 Milvus 内的 vector id。

**Phase 3**:若样板量级增长到几十万,需要考虑 sample 元数据也分库或读副本。

### 3.4 订单数据

**1A**:不接入。

**Phase 2**:业务人员端**不落库**,通过老板端 API 实时查询。`/api/v1/customers/{id}/orders` 后端调用老板端 `/api/v1/orders?customer_external_id=...`。理由:订单数据量级远大于客户,本地副本管理成本高,实时查询足够支撑销售场景。

## 5. AI 服务集成

### 5.1 DeepSeek 文本 LLM

- **接入方式**:业务人员端后端通过 `httpx.AsyncClient` 直接调 DeepSeek REST API。
- **密钥管理**:`DEEPSEEK_API_KEY` 通过 docker env 注入,**前端永不持有**。
- **错误处理**:超时(30s)、限流(429)走指数退避重试(最多 3 次);最终失败返回固定 fallback 文案("AI 暂时不可用,请稍后再试")。
- **Function Calling**:业务人员端定义 7 个工具(查客户/查拜访/查样板/查订单/生成周报/AI 摘要/找板),发给 DeepSeek;模型决定调哪个工具时,后端真实执行 SQL 查询(经 owner_id 鉴权过滤)并回传结果。
- **流式响应**:`POST /api/v1/ai/chat` 支持 Server-Sent Events(SSE)模式,前端实时渲染输出。
- **1A**:全部 stub,Phase 1B 真接入。

### 5.2 公司视觉服务(SigLIP2/DINOv2 + Milvus)

- **接入方式**:**业务人员端不直接跑视觉模型**——视觉工程师把模型封装为内部 HTTP/gRPC 服务,业务人员端调用。
- **API 契约**(草案,待视觉团队定):
  ```http
  POST https://vision-service.dingwei.internal/api/v1/sample-search
  {
    "image_base64": "...",          # 或 "image_url": "..."
    "description": "提花织带,黑色,20mm 宽,手感偏软",
    "filters": {
      "width_mm_range": [18, 22],
      "ribbon_type": "提花",
      "tension_range": ["中", "中高"]
    },
    "top_k": 10
  }

  Response 200:
  {
    "matches": [
      {
        "external_id": "SAMPLE-2024-HY-0823",
        "sample_no": "HY-20-BK-软手感",
        "score": 0.92,
        "image_url": "https://...",
        "metadata": { "width_mm": 20, "ribbon_type": "提花", ... }
      }
    ]
  }
  ```
- **业务人员端职责**:
  - 接收销售上传的图(via `/api/v1/ai/board-search`),转发给视觉服务。
  - 收到 matches 后,把 `external_id` 映射回本端 `sample` 表(用 `sample.external_id`)获取业务信息(关联客户、上次确认时间等),组装最终响应给前端。
  - 不存储 vector,不跑模型。
- **1A**:stub,Phase 2 与视觉团队对接后真接入。

## 6. 账号体系(悬挂问题)

**用户已确认**:业务人员端和老板端是双系统,各自展示自身内容(业务人员端兼有个人日历用途)。**但未明确选定账号统一方案**。

**1A 阶段(已确定)**:

- 业务人员端**自有账号**,bcrypt 密码,JWT 鉴权。
- `user.external_id` 字段预留,可在 seed 时填入老板端账号 ID,后期同步用。
- 试点期 3-5 个销售,seed 一批账号即可。

**上线前必须对齐的方案选项**(与老板端团队 Phase 1B 早期讨论):

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 老板端做 OIDC IdP** | 销售只记一套账号;权限由老板端集中管 | 老板端需提供 SSO endpoint,重构期可能没空 |
| **B. 老板端定时同步用户清单到业务人员端** | 业务人员端继续独立鉴权,无需老板端配合 SSO | 密码两套(同步密码哈希不安全) |
| **C. 公司 LDAP/AD 作为统一身份源** | 与公司 IT 体系融合 | 需要 IT 团队提供 LDAP,试点期内不一定能搞定 |
| **D. 双系统独立账号 + external_id 映射** | 0 改动,最快 | 销售要登两次账号,体验差,但试点期可接受 |

**推荐**:**Phase 1A 走方案 D**(已实施),**Phase 1B 早期与老板端团队评估方案 A**(理想)或 **B**(妥协),**上线前必须有结论**。

## 7. 接口契约管理

- 业务人员端 FastAPI 自动产 OpenAPI,前端通过 `openapi-typescript` 生成 TS 类型。**这是业务人员端 ↔ 前端的契约工具**。
- 业务人员端 ↔ 老板端的契约采用 **OpenAPI 3 文档 + 双方 review**——Phase 1B 启动时,业务人员端把自己希望从老板端拿到的数据写成 OpenAPI 文档草稿,与老板端团队对齐字段、状态码、错误格式。
- 重大契约变更走 **versioned endpoint**(`/api/v1` → `/api/v2`),不在 v1 上做破坏性变更。

## 8. 错误处理与恢复

### 8.1 同步失败

- 客户主数据拉取失败:后台任务重试 3 次,指数退避;持续失败则在 admin 视图(Phase 2)显示告警,不阻塞前台业务。
- 业务人员端 → 老板端推送失败:`visit_record.sync_failed_at` 字段(Phase 2 添加),保留失败计数,人工介入排查。

### 8.2 数据冲突

- 单向同步设计本身避免大部分冲突。
- 客户基本信息冲突:老板端版本优先(它是 source of truth)。
- 客户 owner 冲突:业务人员端版本优先(因为业务人员端管理 owner)。

### 8.3 灾难恢复

- SQL Server:docker volume 每日备份(脚本由运维提供);生产 SQL Server 实例走公司标准备份策略。
- Milvus(Phase 2):向量数据可从原始样板图重新计算,只需恢复元数据。
- 应用层:无状态(JWT 不存 session,文件存 docker volume 已备份),起新实例即可。

## 9. 安全

- **传输层**:全 HTTPS,Nginx 终结 TLS。内网用公司私有 CA,外网用 Let's Encrypt 或商业证书。
- **认证**:JWT 短期 token(1A 用 24h,Phase 1B 加 refresh token)。
- **授权**:服务层强制 `owner_id` 过滤(见 `docs/需求文档-v2.md` §3),没有任何端点能绕过。
- **API 间认证**(业务人员端 ↔ 老板端 ↔ 视觉服务):service account token,内网 mTLS。
- **敏感数据**:客户电话、报价不写 log;数据库连接字符串、JWT_SECRET、DEEPSEEK_API_KEY 走 docker secrets 或 env vault,不入 git。
- **照片上传**:mime 校验用 magic bytes;5MB 限制;唯一文件名 UUID;访问鉴权(只允许 owner / manager 看);per-user 限速。
- **跨域**:CORS 严格白名单(只允许业务人员端前端域名),不开 `*`。
