## 0. 术语对照(避免歧义)

下列词汇在本文档中常被混用,先界定:

| 词 | 含义 | 易混点 |
|---|---|---|
| **plan** | `plan` 表的一行,**结构化日程项**,有时间/客户/状态字段 | 不是周报里的"下周计划"文本 |
| **next_plan** | 周报 4 段中的"下周工作计划"字段,**自由文本** | 不是 plan 表的某条记录,与 plan 表**不自动同步** |
| **周报 submit** | 销售把 draft 状态周报提交,状态 → submitted | **不是审批**——提交即生效,无需主管批准 |
| **周报 reopen** | 主管把 submitted 状态周报退回,状态 → reopened,销售可重新编辑 | **不是 reject**——是"补充更多内容"而非"拒绝" |
| **客户转移 manager_direct** | 主管发起,**立即生效**,无审批步骤 | 与 sales_request 区分 |
| **客户转移 sales_request** | 销售发起,**需主管 approve/reject** | 唯一的真正"审批"流程 |
| **审批** | 本应用中**唯一有审批的动作**是 sales_request 客户转移 | plan / 周报 submit / 拜访录入都没有审批 |
| **个人 plan / is_personal=True** | 销售自己加的"喝水/站立"等非业务提醒 | 不回传老板端,主管视图自动过滤 |
| **超期** | `last_visit_at` 距离 now() 超过等级阈值的客户 | 不存表,运行时计算 |
| **owner_id** | `customer.owner_id` — 客户当前的负责销售 | 一个客户只有一个 owner |

## 1. 概述

鼎伟 CRM 业务人员端是面向鼎伟织带工厂**销售/业务员**的手机端 CRM 应用。它**不是老板端**——老板端是同期在重构中的姊妹系统,两者数据要打通,但视角、首页、交互完全不同。

本应用的核心承诺:**让销售围绕日历完成日常工作,在沉淀客户数据的同时主动自检客户照顾及时性,并把数据通畅地汇报给老板端**。

### 1.1 不做老板端

老板端有自己的应用、自己的视角(全局看板、汇总报表、团队 KPI)。本应用只服务**销售一线 + 主管中层**两种角色。老板不进本应用。

### 1.2 与老板端的关系

老板端正在重构,与本应用作为**姊妹项目同期上线**。两者数据通过 REST API 单向同步(老板端 → 业务人员端拉客户主数据;业务人员端 → 老板端回传拜访/计划/周报)。具体集成方案见 `docs/系统集成方案.md`。

## 2. 业务驱动 / Why now

通过访谈梳理出的核心动机(按优先级):

1. **客户信息沉淀**——防人员流动导致客户关系流失。资深销售一走客户跟着走,公司想把这种关系沉淀进系统资产。
2. **销售日常高效化**——销售目前用 Excel + 微信群 + 纸笔混合管理拜访计划/周报/客户档案,工具碎片化,信息难检索。
3. **老板要实时数据,但销售录入不起来**——老板端已经在用,但数据靠销售手工补录,质量差。需要一个销售**自愿用**、**轻量好用**的工具,让销售一边干活一边自然产生数据。
4. **给 AI 提效套路提供数据底座**——周报草稿、拜访摘要、客户问答、找板这四个 AI 功能没有结构化数据底座根本做不出来。
5. **销售自检客户照顾及时性 + 与老板及时汇报对接**(本项目的关键差异化诉求,用户原话):
   - 销售自己要能自查"哪些客户超期了没去看",主动避免遗漏。
   - 老板要能透过周报通畅看到销售本周做了什么、下周打算做什么。

## 3. 角色与权限

业务人员端只识别两种角色(老板用老板端,本应用看不到老板角色):

### 3.1 Sales(销售员)

- 占用户绝大多数,**默认角色**。
- 看到的范围:**只属于自己的**客户、拜访、计划、周报。永远不能看到别人的。
- 可以做的动作:
  - 浏览/筛选自己的客户,看客户 360。
  - 在自己的日历上新增/修改/删除拜访计划和个人提醒。
  - 现场录入拜访记录(支持文字 + 照片)。
  - 写周报(可调用 AI 生成草稿)、修改、提交。
  - 调 AI 问答和 AI 找板(1A 全部 stub,Phase 1B/2 真实接入)。
  - 提交"客户转移请求"(报批流程,主管审批后生效)。

### 3.2 Manager(主管)

- 中层主管,有若干 Sales 作为直接下属(`user.manager_id` 关联)。
- **同时也是 Sales**——主管自己也有客户、自己也要拜访、自己也写周报。**调 `/customers` 等基础端点时返回本人数据,不是下属数据**。主管视角通过 `/manager/*` 独立端点访问。
- 额外看到的内容(走 `/manager/*` 端点):
  - 下属销售名单 + 每个下属的本周拜访总数、超期客户数(`/manager/team-summary`)。
  - 指定下属的拜访列表分页(`/manager/subordinates/{userId}/visits`)。
  - 下属提交的"客户转移请求"待审列表(`/customer-transfers?status=pending`),可 approve / reject。
  - 下属的周报只读视图(可退回让下属补充)。
- 可以做的动作:
  - 直接发起"客户转移"(主管直接转,立即生效,无需审批)。
  - 退回下属周报(从 submitted → reopened)。

### 3.3 不存在 Boss 角色

老板若要看全局数据,走老板端。本应用任何角色都不应该有跨主管视角的全局看板。

### 3.4 客户归属

- **一个客户只有一个 owner**(`customer.owner_id` 单值,非数组)。
- **客户转移有两条流程**(都要在 1A 实现):
  - **主管直接转**(`flow='manager_direct'`):主管选客户 + 选新 owner,立即生效。`customer_transfer` 记录 `status='executed'`。
  - **销售报批**(`flow='sales_request'`):销售提交转移请求(指定目标 owner + 理由),主管在待审列表中 approve / reject。approve 时 `customer.owner_id` 更新,`status='approved'` → `executed`。**1A 销售不能撤销已提交的请求**。
- 客户的历史拜访记录、计划、周报里关于该客户的内容**不随 owner 变动而隐藏**——转移后新 owner 看得到完整历史,原 owner 不再看到该客户但其历史拜访仍归原 owner 名下。

### 3.5 角色 × 功能矩阵(Phase 1A 实施依据)

> 本节是 Phase 1A 编码的"角色对齐表"——后端写 router 时按这份对照"哪些字段过滤""哪些角色才能调",前端写组件时按这份对照"哪些组件只渲染给 Manager"。详细 Sales/Manager 页面级走查见 `/Users/jasper/.claude/plans/expressive-snuggling-crystal.md` §2-3。**ASCII 速查表见本文档附录 C**。

#### 3.5.1 设计原则

1. **Manager 是 Sales 的超集**——拥有 Sales 全部功能,**额外**增加"主管视角"。`role='manager'` 调 sales 端点按本人数据走;调 manager-only 端点才进入主管视角。
2. **数据归属严格隔离**——Sales 永远只能看 `owner_id = current_user.id` / `salesperson_id = current_user.id` 的数据。
3. **Manager 看下属是受控的**——通过 `user.manager_id` 关联,只看**直接下属**,看不到其他主管的下属。
4. **没有全局视角**——任何角色都拿不到"全公司所有客户"列表。全局看板走老板端。

#### 3.5.3 数据可见性硬规则(service 层 SELECT 子句必加过滤)

| 资源 | Sales / Manager 本人 | Manager 看下属 |
|---|---|---|
| `plan` | `WHERE salesperson_id = :uid` | `WHERE salesperson_id IN (SELECT id FROM user WHERE manager_id=:uid) AND is_personal = 0` |
| `customer` | `WHERE owner_id = :uid` | 不直接查客户列表,通过 visit_record 反查 |
| `visit_record` | `WHERE salesperson_id = :uid` | `WHERE salesperson_id IN (SELECT id FROM user WHERE manager_id=:uid)` |
| `weekly_report` | `WHERE salesperson_id = :uid` | `WHERE salesperson_id IN (...) AND status IN ('submitted','reopened')` |
| `customer_transfer` | `WHERE initiated_by_user_id = :uid` | 待审:`WHERE flow='sales_request' AND status='pending' AND from_user_id IN (SELECT id FROM user WHERE manager_id=:uid)` |
| `visit_attachment` | 通过 `visit_record.salesperson_id` 反查 | 同上链路 |

**实施位置**:每个 service 方法首参 `user: AuthUser`,where 子句必带 owner/sales 过滤。Code review 时全文 grep `select(` / `query(`,核对是否漏掉过滤。

#### 3.5.4 边界 case(实施时要注意)

1. **Manager 本人也是 Sales**:Manager 调 `/customers` 返回本人客户而非下属。Manager 视角是**单独端点**(`/manager/*`),不污染基础端点。
2. **客户转移后的历史**:`visit_record.salesperson_id` 不变(历史归原 Sales),`customer.owner_id` 改。新 owner 看 360 时展示完整时间线(含原 owner 的拜访)。
3. **is_personal 误判**:Sales 在 custom plan 无 customer 时**默认 personal=True**,前端必须给 toggle 让 Sales 创建时明确"个人提醒 / 工作准备"(1A 必做)。
4. **Manager 看下属照片**:鉴权链路 `visit_attachment → visit_record.salesperson_id → user.manager_id == current_user.id`。链路稍长,写单元测试。
5. **退回的周报重新提交**:reopened → submitted 1A 不记录提交次数。
6. **客户转移并发**:同一客户被两个销售同时转。1A 用乐观锁 `UPDATE customer SET owner_id=:new WHERE id=:cid AND owner_id=:old_owner`。
7. **next_plan 与 plan 表不同步**:周报 next_plan 是自由文本,plan 表是结构化日程。AI 草稿(Phase 1B)读 plan 表合成 text 写进 next_plan,但销售改 next_plan 不反向改 plan。详见 §5.5。

## 4. 核心业务流程

### 4.1 销售的一天

(原型保留双主题:**日程优先**和**月历优先**,销售自己选)

- **开机第一眼**:进入日历页,默认看到今天的拜访计划列表。**首屏/日历上有主动提醒**——比如"今天有超期客户 3 个,点击查看"——不依赖客户页筛选的被动呈现。
- **拜访前**:点开计划看客户简要信息和上次拜访摘要。
- **拜访现场**:实时录入拜访记录:**文字 + 照片**(语音延后)。客户拒接/没空也算一次接触,可录"未达成"状态。
- **拜访后**:可让 AI 把粗记摘要成业务总结(Phase 1B 真接入,1A stub)。
- **添加个人提醒**:可以在日历上加"喝水/站立/午休"等非业务提醒(`is_personal=True`),这些不回传老板端,主管视图不可见。
- **遇到客户问类似产品**:打开 AI 找板,上传一张图 + 填工艺参数(宽度/开力度/颜色等),系统返回相似度排序的历史样板(Phase 2 真接入)。
- **下班前**:看一眼超期客户数,决定明天行程。

### 4.2 销售的一周(周报语义)

- 周中正常拜访录入,自然产生数据。
- 周五/周六(理论上任何时点都可,但推荐周末):打开周报页 → 点"AI 生成草稿"(1A stub,Phase 1B 真接入)→ 系统基于**当周**(本周一到现在)的拜访记录 + plan + 样板变化 → 生成草稿 → 销售人工修改 → 点"提交"。
- **关键澄清**:
  - 周报覆盖范围是 **当周**(正在结束的这一周),**不是上一周**。一个销售一周一份(`UQ_weekly_owner_week` 唯一约束)。
  - 周报 4 段固定结构:
    1. `summary` — 本周工作总结
    2. `next_plan` — 下周工作计划(自由文本,**与 plan 表不自动同步**,详见 §5.5)
    3. `notes` — 备注事项
    4. `attachments` — 附件(1A 字段保留,不开上传 UI)
  - 提交 = 状态 `draft → submitted`,**提交即生效,主管能立即看,无需审批**。
- 提交后主管可读,可点"退回"让销售补充(`submitted → reopened`)。退回**不是拒绝**——是让销售补内容,改完销售再提交一次。

### 4.3 主管的额外动作

- 早上看一眼 `/manager/team-summary`:下属本周拜访总数、超期客户数,识别需要 1:1 的同事。
- 处理待审的客户转移请求(`/customer-transfers?status=pending`)→ approve / reject。
- 周末读下属周报,必要时退回补充。
- 必要时主动发起客户转移(`flow='manager_direct'`,立即生效)。

### 4.4 拜访照片上传

- 现场拍照,前端调 `POST /uploads/visit-photo`(multipart,5MB 限,JPEG/PNG),拿到 `attachment_id`。
- 提交拜访记录时把 `attachment_id` 数组放在 `POST /visit-records` 的 body 里。
- 照片存储路径:`uploads/visits/{yyyymm}/{uuid}.jpg`(本地 docker volume,1A 起步;Phase 2 迁 MinIO/OSS)。
- 安全:
  - mime 校验用 **magic bytes** 而非扩展名;
  - 访问鉴权(只允许 owner / 该 owner 的 manager 看);
  - per-user 上传限速(数值待 Phase 1A 实施时定,建议初值 30 张/小时 + 100 张/天);
  - 文件名 UUID,防穷举。

### 4.5 状态机汇总

所有状态机一处集中,实施时按此对照。

```
plan (日程)
  pending ──销售点完成──▶ done
  pending ──销售点取消──▶ cancelled
  (创建立即 pending,不经审批)

weekly_report (周报)
  draft ──销售 submit──▶ submitted ──主管 reopen──▶ reopened
                            ▲                          │
                            └──销售再 submit───────────┘
  (submit 无审批,reopen 非拒绝)

customer_transfer (客户转移)
  ─ flow='manager_direct':
      pending ─── 后端立即执行 ──▶ executed
      (实际上一步到位,不在 pending 停留)
  ─ flow='sales_request':
      pending ──主管 approve──▶ approved ──后端执行──▶ executed
      pending ──主管 reject ──▶ rejected
      (1A 销售不能撤销 pending)

customer (客户)
  active / inactive  (1A 由老板端同步,不在本应用 CRUD)

user (用户)
  active / disabled / resigned  (1A 由 seed 写入)

visit_record (拜访)
  无状态机 — 一次性写入,可关联多张 visit_attachment

sample (样板)
  qiyang/zhongban/queren/yizhuandingdan/yifangqi
  (1A 仅建表保留字段,无 router)
```

## 5. 数据契约

### 5.1 客户分级 A/B/C

- 1A:**默认 B**,字段保留,销售可手动 override。
- Phase 2:**AI 估评**——根据拜访频次、意向状态、样板进度综合打分,自动调级(写入 `customer.ai_score`)。销售仍可手动 override `customer.level`。
- 不是按销售额硬切——这家工厂的销售模式里"客户重要性"是综合的、AI 友好的判断。

### 5.2 超期判定规则

- A 类客户 **14 天**未拜访 = 超期。
- B 类客户 **30 天**未拜访 = 超期。
- C 类客户 **60 天**未拜访 = 超期。
- 判定逻辑:**运行时计算**(`customer.last_visit_at` 与 `now()` 比较),不存表。
- 首屏/日历显示"今天超期 X 个"靠 `GET /customers/overdue-summary`,返回数字 + 列表前 5。

### 5.3 与老板端共享 ID

为后续打通预留 `external_id` + `source_system` + `synced_at` 三字段。1A 不写同步代码,只建字段:

- `user.external_id` — 销售人员 ID 对接老板端账号
- `customer.external_id` — 客户主数据代号(老板端是 source of truth)
- `sample.external_id` — 样板号
- `order` 实体(1A 不建表)— 后续 Phase 2 加,带 external_id

业务人员端**产生的数据**(plan / visit_record / weekly_report / customer_transfer)回传时用本端 id 作为唯一标识,不需要 external_id。

### 5.4 个人 vs 业务计划

- `plan.is_personal BIT NOT NULL DEFAULT 0`
- 创建时**后端推断默认值**:`type='custom' AND customer_id IS NULL` 默认 `True`(纯个人提醒);其他默认 `False`(业务计划)。
- 销售可在前端表单上 toggle 手动改(预填后端推断结果)。
- **影响**:
  - `is_personal=True` 的 plan **不回传老板端**(Phase 2 集成时过滤)。
  - 主管视图(team-summary / subordinate visits)**自动过滤掉 `is_personal=True` 的项目**——主管只看下属的业务计划,不窥探下属的喝水提醒。

### 5.5 周报字段与 plan 表的关系(易混)

这是一处实施时最容易踩坑的地方,显式约定:

| | plan 表 | weekly_report.next_plan 字段 |
|---|---|---|
| 类型 | 结构化(一行一日程,有时间/客户/状态) | 自由文本(textarea) |
| 写入者 | 销售在日历 tab 新增 | 销售在周报 tab 编辑 / AI 草稿生成 |
| 时间范围 | 任意(下周、下月、过去都可) | 仅描述下一周计划 |
| 状态 | pending / done / cancelled | 无状态 |
| 同步关系 | **单向**:AI 草稿(Phase 1B)读取销售下周已存在的 plan → 合成 text → 写入 next_plan | 销售在周报里改 next_plan 文本**不反向**改 plan 表 |
| 回传老板端 | `is_personal=False` 的回传 | 整份周报回传(包含 next_plan 文本) |

**含义**:周报里"下周计划"自由文本是面向"老板看周报"的叙事体,plan 表是销售自己用的日历日程。两者各自服务不同读者,不必强行一致。

## 6. AI 功能(本期 1A 全部 stub,接口契约固定)

四个 AI 功能均是真痛点(经访谈确认,无 demo 想象项):

### 6.1 AI 周报草稿

- 销售点"AI 生成草稿" → 后端聚合本周拜访记录 + plan + 样板变化 → DeepSeek 文本 API 生成符合周报 4 段结构的草稿。
- 1A 是 stub,返回固定文本;Phase 1B 真接入。
- 输入聚合:当周 visit_record + 当周 plan(完成 + 未完成)+ 当周 sample 变化(若已实现)。
- 输出:`{ summary, next_plan, notes, attachments=[] }` 直接预填周报表单。

### 6.2 AI 拜访摘要

- 销售粗记的口语化拜访内容 → AI 摘要成业务总结(意向、跟进点、风险)。
- 1A stub,Phase 1B 真接入。

### 6.3 AI 客户问答

- 销售在 AI 页输入"宏远最近订单和拜访情况" → 后端通过 Function Calling 查 customer / visit / order 等结构化数据 → DeepSeek 回话。
- 1A stub,Phase 1B 真接入。

### 6.4 AI 找板(以图搜图 + 文本 + 工艺参数)

**业务场景**:客户问"你们有没有类似的样品",销售上传一张图 + 描述需求,系统从公司**已积累的几万张样板图**库中找最相似的历史样板。

**算法设计**(Phase 2 实施,1A 仅 stub):

- 视觉编码:**SigLIP2**(优于 CLIP,细粒度 + 中文好)+ **DINOv2**(纯视觉路径,细粒度纹理强)双路 late fusion。
- 文本对齐:SigLIP2 的 text encoder 处理"提花 / 颜色 / 宽度 / 手感"等描述。
- 向量库:**Milvus**(几万张以上用 Milvus,Qdrant 在万级以下更轻)。embedding 不进 SQL Server(SQL Server 2025 向量类型尚预览,不稳)。
- 检索流程:文本 + 图 → late fusion embedding → ANN top-K → metadata filter(width range / ribbon_type / 开力度 等)→ top 10 返回。
- **质量关键**:用公司样板图做 LoRA 微调,通用预训练对垂直领域零样本效果有限。需要有经验销售/技术工程师标 200-500 组"相似/不相似"配对做 contrastive 微调和评测。

**部署提醒**:SigLIP2/DINOv2 base ~500MB,GPU 推理 50ms/张,CPU 推理不可行,需公司 GPU 服务器或 GPU 云。

**1A 表现(Q6 决议)**:UI 完整(客户场景/图片/宽度/开力度/织带类型/颜色要求 6 字段),图片真上传走 `POST /uploads/visit-photo`,提交调 `/ai/board-search` stub 返回 3 条拟真 mock 样板。试点销售能完整体验交互,反馈才有意义。

## 7. 非功能性需求

- **性能**:试点期支持 50 同时在线(中型极限),不加缓存层,加必要索引(见数据字典)。
- **部署**:**多部署**——内网 + 外网销售手机可访问。需要 HTTPS 反向代理,1A 文档化即可,生产前配置 TLS。
- **移动网络**:销售在工厂/客户现场可能 4G 信号弱,API 设计要轻量,大数据传输(照片上传)允许分块。**1A 不做离线模式**——网络断开时用户感知失败,不做本地草稿。Phase 4 再加。
- **审计/留痕**:`weekly_report` 状态变更、`customer_transfer` 流转、`customer.owner_id` 变更需要落审计日志。1A 不强制,Phase 2 加。
- **安全**:
  - 密码 bcrypt 哈希,seed 默认 `123456`(试点前需评估是否改强密码)。
  - JWT 鉴权,token 写在 Authorization header,不写 localStorage 的明文 cookie。
  - 拜访照片访问需 JWT 校验,且鉴权链路在 service 层显式校验。
  - mime 校验走 magic bytes,不信任前端扩展名。

## 8. 集成边界

### 8.1 账号体系

- **1A**:业务人员端**自有账号**,bcrypt 密码,JWT 鉴权。`user.external_id` 字段预留。
- **悬挂问题**:用户没有明确选定 SSO 路径(OIDC vs LDAP vs 数据同步)。**上线前必须与老板端团队对齐**统一身份方案,否则销售要登两次账号是体验灾难。
- 1A 阶段 seed 一批账号(zhangwei / liuyang / chenmin / wangManager),试点用。

### 8.2 客户主数据同步

- **方向**:老板端 → 业务人员端(单向)。
- **方式**:老板端提供 REST API,业务人员端定时拉取(daily / 增量)。
- **1A**:不实施同步,仅预留字段。Phase 2 与老板端团队对接后实施。

### 8.3 业务人员端产生数据的回传

- 拜访记录、计划(只回传 `is_personal=False` 的)、周报、客户转移记录需要回传到老板端,供老板端报表用。
- **1A**:不实施回传,数据仅在业务人员端 DB。Phase 2 实施。
